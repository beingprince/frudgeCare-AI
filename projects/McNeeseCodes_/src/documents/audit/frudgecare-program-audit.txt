1. EXECUTIVE SUMMARY
================================================================================

A. HUMAN LANGUAGE
-----------------
FrudgeCare AI is an "AI-orchestrated care coordination platform" — a web
app that takes a patient's self-reported symptoms, lets an AI propose how
urgent the case is, then walks the case through the hands of a Front Desk
clerk, a Nurse (who must validate everything the AI suggested), and finally
a Doctor/Provider who signs the final decision. Operations staff can watch
the whole thing through a dashboard and an immutable audit log.

The code is real and mostly works today in a DEMO mode:
  - You can log in as any of four personas on the landing page and move
    through the screens.
  - The AI part (FastAPI microservice that calls Google Gemini) is wired
    up and called from the Next.js app.
  - Data is either read from an in-memory mock data set (12-20 realistic
    cases) or, if Supabase env vars are set, from a real Postgres/Supabase
    database using the migrations in supabase/migrations/.

Two login flows exist:
  - Patient: extremely low-friction — phone OR email + date of birth. No OTP.
  - Staff  : a 5-step multi-factor flow (staff code + username -> OTP ->
             email + password), with role-to-panel integrity checks.

The system is feature-complete at the "hackathon demo" level, but there
are tracked design-debt items (spec 21-known-design-debt.md), a production
auth path that still relies on placeholders (bcrypt + Twilio), and many UI
flows use mocked data rather than real DB inserts. QA-wise the screens are
consistent, accessible, and follow a documented design system, but the
nurse -> provider handoff contract, while well specified, is not fully
enforced end-to-end (see § 7.3 and § 14).

B. TECHNICAL
------------
- Monorepo (npm workspaces): apps/web (Next.js 16, React 19, TS, Tailwind,
  MUI v9), services/ai-engine (FastAPI + google-genai), supabase (raw
  migrations + seed), packages/* (reserved, empty today).
- Two brand colors, two token systems (CSS vars + MUI `C`) — flagged debt.
- Auth: jose (JWT HS256), httpOnly cookie `fc_session`, 8h staff / 12h
  patient expiry, route prefix guard in `apps/web/src/proxy.ts`.
- Demo bypass: `NEXT_PUBLIC_DEMO_MODE=true` disables middleware guard and
  accepts canned credentials (see § 5, Appendix A).
- Data path: `lib/supabase.ts` + `lib/mock-service.ts` (dual-path, same
  shape).
- AI path: `/api/ai/analyze-intake` proxies to
  `http://localhost:8001/analyze-intake` on the FastAPI service.
- State machine (spec 18) is implemented in part: nurse page enforces
  handoff checklist, provider page has a "not-cleared gate", but audit
  event `nurse.handoff_ready` is still a TODO (only a browser alert today).

2. PRODUCT VISION & ROLES
================================================================================

A. HUMAN LANGUAGE
-----------------
The platform presents itself as a "clinical traffic controller":
  - Patient writes symptoms -> AI reads them -> suggests urgency and a
    clinical brief.
  - Front desk accepts or overrides urgency, assigns providers, books a
    slot.
  - Nurse confirms the AI's story is actually true (vitals, questionnaire,
    red flags, AI flags acknowledged) BEFORE the doctor sees anything.
  - Doctor makes the final call (prescribe, order test, refer, discharge).
  - Everybody's action is written to an audit log.

Roles:
  Patient       – enters symptoms, watches status
  Front Desk    – traffic control, urgency override, scheduling
  Clinical Nurse– the coordination + data-validation gatekeeper
  Doctor        – final clinical authority
  Admin/Ops     – system and analytics oversight
  AI Orchestr.  – generates drafts, never decides alone (human-in-the-loop)

B. TECHNICAL
------------
Canonical source: documents/product/working flow and vision of the system.txt
Role enum (TS)    : apps/web/src/types/index.ts
    export type UserRole =
      'patient' | 'front_desk' | 'provider' | 'operations' | 'admin';
Role enum (SQL)   : supabase/migrations/20240417000000_initial_schema.sql
    create type public.user_role as enum
      ('patient','front_desk','provider','operations','admin');
Note: A "nurse" role does NOT exist as a first-class DB/TS enum. The /nurse
page is implemented and the workflow spec treats "Nurse" as a named role,
but the DB role enum and the route guard do not recognize it. This is a
real gap (see § 14, finding F-01).

3. TECHNOLOGY STACK & RUNTIME ARCHITECTURE
================================================================================

A. HUMAN LANGUAGE
-----------------
Three moving parts run side-by-side:
  1. A Next.js web app (port 3000) — the thing users see + API routes.
  2. A tiny Python FastAPI service (port 8001) — asks Google Gemini for the
     AI triage JSON. It is a separate service on purpose so the web app
     doesn't talk directly to Google.
  3. Supabase Postgres (optional) — when configured, all real data lives
     here. Without it, the app falls back to the in-memory mock dataset.

The web app runs auth, UI, AI proxy, logout, and a demo-reset endpoint.
The Python service runs AI and two "orchestrator" endpoints that enforce
invariants (e.g. you can't mark a case 'provider_review_pending' unless a
nurse explicitly validated it).

B. TECHNICAL
------------
Root package.json workspaces: ["apps/*", "services/*", "packages/*"]
Run scripts:
    npm run dev        -> concurrently runs web + ai
    npm run dev:web    -> next dev on :3000
    npm run dev:ai     -> uvicorn main:app --reload --port 8001
    npm run build      -> next build
    npm run lint       -> next lint

Web stack:
  - Next.js (app router) — see documents/engineering/original_root/AGENTS.md:
    "This is NOT the Next.js you know. This version has breaking changes
    — read the relevant guide in node_modules/next/dist/docs/".
  - React 19 + TypeScript (strict).
  - Tailwind v4 utility CSS + MUI v9 (Material UI) coexistence.
  - framer-motion for motion. lucide-react + @mui/icons-material for icons.
  - jose for JWT. @supabase/supabase-js for DB. @google/generative-ai in JS
    wrapper (lib/gemini.ts) — currently unused; real calls go to Python.
  - recharts for operations dashboard charts.

AI service stack:
  - FastAPI + pydantic + google-genai (newer SDK; imports
    `from google import genai`).
  - CORS open to http://localhost:3000.
  - Env: services/ai-engine/.env provides GEMINI_API_KEY.

Cross-service contract (what the web calls):
  POST /analyze-intake    { symptoms, duration, severity, patient_history? }
  returns AIAnalysis { urgency, summary, risks[], reasoning, clinician_brief }

Two orchestrator-style endpoints (exist, not currently called from web):
  POST /orchestrator/handoff-to-provider
  POST /orchestrator/submit-provider-action

6. AUTHENTICATION & SESSION LOGIC
================================================================================

6.1 PATIENT LOGIN
-----------------
A. HUMAN LANGUAGE
A patient types their phone OR email plus date of birth. In demo mode,
any combination is accepted as "John Miller" and the app drops them at
/patient/intake. In real mode, the backend queries patient_profiles, checks
the DOB matches, and drops them at /patient/status.

B. TECHNICAL
Route : apps/web/src/app/api/auth/patient/login/route.ts
Input : { phone_or_email, date_of_birth }
Demo  : issues JWT with role='patient', cookie maxAge = 12h,
        redirect '/patient/intake'.
Real  : supabase.from('patient_profiles')
          .or('email.eq.<val>,phone.eq.<val>').single();
        verifies date_of_birth; issues JWT; redirect '/patient/status'.
UI    : apps/web/src/app/auth/patient/page.tsx (mobile-first dark/light shell).

6.2 STAFF LOGIN — 5-STEP MFA
---------------------------
A. HUMAN LANGUAGE
Staff go through five gates:
  Step 1: Staff code (e.g. FD-001) + username (e.g. maria).
  Step 2: System confirms identity and auto-sends an OTP to the phone on
          file (masked display).
  Step 3: User enters the 6-digit OTP. In demo, it's literally 000000.
  Step 4: User enters email + password. In demo, email must match the
          account on file and password must be "demo1234".
  Step 5: System cross-validates all 5 factors AND checks the "panel"
          (admin/front-desk/provider) matches the account's actual role.
          If anything mismatches -> 403 + audit log.
Success sets an httpOnly JWT cookie (fc_session) and redirects to the
role's home page.

B. TECHNICAL
UI           : apps/web/src/app/auth/staff/[panel]/page.tsx
Step 1+2 API : /api/auth/staff/verify-identity
               - lookupDemoStaff(staff_code, username) OR Supabase query.
               - returns user_id, role, name, phone_masked.
               - checks `active` + `locked_until` (rate limit).
Step 3 API   : /api/auth/staff/send-otp
               - generateOTP() -> in demo always "000000"; logs to console
                 via sendOTP().
               - real mode: insert otp_sessions row, call Twilio (TODO).
Step 4 API   : /api/auth/staff/verify-otp
               - demo: hardcoded accept of "000000", returns random
                 32-byte hex verification_token.
               - real: look up latest matching otp_sessions row, check
                 expires_at, set verified=true + verification_token.
Step 5 API   : /api/auth/staff/login
               - demo: find account by user_id in DEMO_STAFF_ACCOUNTS, check
                 email, check password via verifyPassword (demo path
                 accepts "demo1234"), check role === expected_panel,
                 else 403 with role_mismatch_attempt audit event.
               - real: supabase query eq(id).eq(email).eq(staff_code);
                 verifyPassword; role check; inserts staff_login_attempts
                 row on success/failure.
               - Sets cookie `fc_session` via response.cookies.set with
                 httpOnly, sameSite=lax, secure in production, maxAge 8h,
                 path '/'.

6.3 SESSION (JWT) LIFECYCLE & COOKIE
-----------------------------------
A. HUMAN LANGUAGE
After any successful login the server creates a signed token that contains
who you are and what role you have. It's stored in an httpOnly cookie so
JavaScript can't steal it. Every protected page request goes through a
middleware that checks that cookie.

B. TECHNICAL
Creation :   createSession() in lib/auth.ts
             -> jose.SignJWT with HS256, 8h expiry, fields:
                { userId, role, name, email, staffCode? }.
Cookie   :   name='fc_session', httpOnly, sameSite='lax',
             secure=(NODE_ENV==='production'), path='/'.
Read     :   getSession() in lib/auth.ts:
                cookies().get('fc_session')?.value -> jose.jwtVerify()
                -> SessionPayload | null.
Destroy  :   destroySession() + /api/auth/logout POST.

A. HUMAN LANGUAGE

7. END-TO-END CLINICAL WORKFLOW
================================================================================

7.0 ONE-PARAGRAPH NATURAL LANGUAGE VERSION
------------------------------------------
A patient opens the app, fills out a four-step intake wizard (basics,
symptoms, scheduling preferences, review) and submits it. The app calls
the AI engine, which returns an urgency level, a clinician brief, a
patient-facing summary, and a list of risk flags. That submission becomes
a "case" with status 'submitted'. The front desk sees it in the queue,
can override urgency, assigns a provider, and books an appointment slot.
The case moves into nurse triage: a nurse fills out a questionnaire,
captures vitals (or explicitly says they weren't taken), confirms the
AI draft brief is accurate, and ticks every item in a validation
checklist. Only then can they "Send to Provider". The provider sees a
three-rail clinical workspace (context | summary | decision), picks a
next action (order test / prescribe / refer / close), writes an encounter
note, optionally writes a patient-visible update, and signs. The patient
sees status, action items, and timeline update. Operations watches
throughput and audits overrides.

7.1 PATIENT INTAKE -> AI PRE-TRIAGE
-----------------------------------
A. HUMAN LANGUAGE
Patient fills out the four-step wizard. Step 2 is the clinically important
one: chief complaint, severity (1-10 segmented buttons), context. On
submit, an AI engine reads the text and returns:
   urgency (high/medium/low), summary (patient-facing), risks[],
   reasoning, clinician_brief (dense 2-3 sentence provider brief).

B. TECHNICAL
UI      : apps/web/src/app/patient/intake/page.tsx (client, 4 steps)
AI Call : POST /api/ai/analyze-intake
           -> proxies body to AI_ENGINE_URL (default
              http://localhost:8001/analyze-intake)
AI Svc  : services/ai-engine/main.py analyze_intake() calls Gemini
          (model=gemini-1.5-flash, response_mime_type=application/json)
Contract: IntakeRequest { symptoms, duration, severity, patient_history? }
          AIAnalysis   { urgency, summary, risks[], reasoning,
                         clinician_brief }
NOTE    : The current intake UI (intake/page.tsx) does NOT call
          /api/ai/analyze-intake; it only handles step navigation and does
          not yet persist a case or invoke the AI. See F-03.

7.3 NURSE VALIDATION HANDSHAKE (THE CONTRACT)
---------------------------------------------
A. HUMAN LANGUAGE
This is the most important workflow boundary in the whole product. The
nurse MUST explicitly tick everything before the "Send to Provider"
button unlocks. The items are:
  (1) Identity verified (pre-populated).
  (2) Chief complaint + severity captured.
  (3) Vitals captured OR explicit "not taken" reason.
  (4) Associated + denied symptoms recorded.
  (5) At least one finding authored OR "no abnormal findings" toggle.
  (6) High-risk AI flags acknowledged.
  (7) Handoff brief reviewed and explicitly confirmed (the AI draft is
      not allowed to pass silently).
The rule is codified in documents/ux design/18-workflow-ownership.md
§ 3.2 and is enforced client-side in apps/web/src/app/nurse/page.tsx.

B. TECHNICAL
File             : apps/web/src/app/nurse/page.tsx
Requirements[]   : REQUIREMENTS array (7 items) — each with passes(data,
                   flags) predicate and a `hint`.
Derived state    : `checklist` via useMemo; allPass = every(done).
Send gate        : handleSend() noop when !isValidated || !allPass.
Side effect gap  : On success, currently displays a browser alert(

7.4 PROVIDER REVIEW / DECISION / RECEIPT
----------------------------------------
A. HUMAN LANGUAGE

15. SECURITY OBSERVATIONS
================================================================================

S-1 JWT
    OK: HS256 + httpOnly cookie + sameSite=lax + secure-in-production.
    Risks: dev-default secret (F-15); no CSRF token for state-changing
    POST endpoints beyond sameSite. Consider double-submit cookie or
    origin check for sensitive endpoints.

S-2 OTP
    Demo OTP is printed to console and returned in the response body
    (demo_otp field). This is explicitly demo-only but can leak if
    NEXT_PUBLIC_DEMO_MODE ever ships to prod.

S-3 Passwords
    password_hash seed is '$2b$10$demo_hash_placeholder' which is not a
    valid bcrypt hash. verifyPassword falls back to `return false` when
    not in demo. Must be replaced with real bcrypt + rehash-on-login
    logic before production.

S-4 Rate limiting
    Schema + constants exist; enforcement incomplete (F-16).

S-5 RLS
    Enabled but minimally policied (F-13).

S-6 FastAPI
    Unauthenticated endpoints (F-14).

S-7 CORS
    AI engine: allow_origins=['http://localhost:3000'] — OK for local.
    Widen only with env-driven list.

S-8 Logging
    Auth events logged to console in demo. Ensure audit_log writes in
    real mode cover all of: identity_failed, role_mismatch_attempt,
    session_destroyed.

S-9 Input validation
    Patient login uses a raw supabase `.or(…)` string interpolated with
    user input — low risk because Supabase-JS PostgREST string filters
    are parameterized, but prefer `.eq()` builder to avoid accidental
    PostgREST injection via operator confusion.

S-10 Demo reset endpoint
    /api/demo/reset has no auth; if shipped to prod with DEMO_MODE off
    it would still run (it doesn't check DEMO_MODE). Guard it.

P-1 Supabase race timeout
    /front-desk/case/[id] and /provider/daily wrap supabase queries in
    a Promise.race against a 500ms timeout and fall back to mock. This
    is a great demo-resilience pattern but will mask real slowness in
    production. Replace with loader states + backoff.

P-2 Next.js turbopack dev
    apps/web/.next/dev tree shows turbopack is enabled; build-manifest
    and transform chunks are large. Not a concern for dev; ensure
    .gitignore covers .next (it does, via standard Next.js template).

P-3 Client-heavy pages
    Most pages are "use client" + framer-motion. Consider server
    components for the shell and for pages that don't need JS state
    (history, audit) to reduce bundle.

P-4 Mock data is in the bundle
    lib/mock-service.ts ships to the client. For production, either
    mark with `if (process.env.NODE_ENV === 'development')` guards or
    move behind /api/demo/*.

D. GLOSSARY
  Case            — A single patient concern opened by intake. Has a
                    case_code (FC-C-####) and flows through the state
                    machine.
  Intake          — Patient-authored symptom submission.
  AI pre-triage   — AI-generated urgency + brief + risks done before
                    any human review.
  Handoff brief   — Nurse-confirmed text that travels to the provider.
  Validation
  checkpoint      — The 7-item gate in /nurse that unblocks "Send to
                    Provider".
  Provider
  decision        — Provider's signed next-action on a case
                    (ProviderAction in DB).
  Demo mode       — `NEXT_PUBLIC_DEMO_MODE=true`. Enables canned
                    credentials and bypasses the middleware route guard.
  Proxy.ts        — This Next.js version's edge middleware entry (the
                    AGENTS.md warns Next's APIs differ).
  fc_session      — httpOnly JWT cookie used for auth across all roles.
  RLS             — Supabase / Postgres Row-Level Security.
