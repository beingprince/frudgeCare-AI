# Responsible AI


This document covers the four areas the organisers required of every team. Data sources, model choices, bias considerations, and failure cases. A short regulatory awareness section is added at the end because the participant guide flagged regulatory awareness under the feasibility scoring criterion.


This is a hackathon prototype. It is not a medical device. It has not been clinically validated. It must not be used to make decisions about real patients.




## 1. Data Sources


### Patient data


All demo patients are synthetic. They were generated by Synthea, an open source synthetic patient generator from MITRE. The pre generated FHIR R4 bundles ship in `services/ai-engine/knowledge_base/synthea_*.json`.


No real patient data was ever loaded. No protected health information has touched the codebase, the AI engine, or the language model context window at any point during development or the demo.


No de identification step is needed because the data is synthetic by construction.


### Clinical knowledge corpus used for retrieval


Twelve short paraphrased clinical heuristics live inline in `services/ai-engine/main.py` under `CLINICAL_GUIDELINES`. They cover qSOFA style sepsis screening, NIHSS style stroke checks, paediatric fever red flags, acute coronary syndrome triggers, and similar widely published patterns.


These are teaching examples written for the demo. They are not authoritative clinical guidelines. The user interface says so on the RAG evidence card.


A production version would replace this with a versioned, clinician curated knowledge base with full citation tracking against PubMed, the CDC clinical guidelines library, NIH MedlinePlus, and FDA DailyMed. All four sources are listed in the participant resource guide.


### Reference terminologies


The engine maps extracted entities to standard codes where possible. The reference files are static lookups, not external API calls.


* ICD-10-CM for diagnosis candidates in `services/ai-engine/knowledge_base/icd10_codes.json`
* Vital sign reference ranges for the warning and critical thresholds in `services/ai-engine/knowledge_base/vitals_ranges.json`
* RxNorm style medication aliases used by the regex extractor


The team did not integrate live SNOMED CT, LOINC, RxNorm, or NDC lookups. All four would belong in a production version.




## 2. Model Choices


### Language model


Google Gemini 2.5 Flash Lite via the `google-generativeai` SDK.


Why this model. It has a higher free tier daily quota than the standard Flash variant. The standard Flash variant ran out of quota during smoke testing on day one. Latency is also lower, which matters for the live demo.


No fine tuning was performed. The team controls prompts and post processing only. The model itself is hosted inference.


The prompt is constrained. It asks for entity extraction, negations, risk flags, and structured JSON. It does not ask for free form clinical advice or treatment recommendations.


### Tiered cascade


The engine runs four tiers in sequence. Every response carries a `source_tier` field so the user interface can show which tier answered.


| Tier | What it is | When it runs | How it appears to the operator |
| ---- | ---------- | ------------ | ------------------------------ |
| 0 | Deterministic regex and keyword routing | Always. Used by the command palette for navigation | Green Tier 0 badge |
| 1 | RAG retrieval over the inline guideline corpus | Always | Returned in the `rag_evidence` block |
| 2 | Gemini language model | When the API key is set and the call succeeds | Violet Tier 2 badge |
| 3 | Safe deterministic fallback | When tier 2 is unavailable or returned nothing usable | Amber Tier 3 badge |


The cascade fails open. A missing API key, a network error, or a quota refusal all cause the engine to fall back gracefully without breaking the user interface.


### Smaller, task specific models the team considered


The participant guide highlighted a number of biomedical and clinical NER models on Hugging Face. The team did not integrate any of them in the hackathon window. A production version of FrudgeCare AI would likely use one of the following for the entity extraction step instead of, or in addition to, the language model.


* `microsoft/BiomedNLP-BiomedBERT-base-uncased-abstract` for biomedical text classification
* `johnsnowlabs/JSL-MedS-NER-q16_v2` for medical named entity recognition
* `aaditya/Llama3-OpenBioLLM-70B` for clinical text reasoning


Specialised models would be more accurate, more reproducible, and easier to evaluate for bias than a general purpose language model.




## 3. Bias Considerations


The team was honest with itself about what it did and did not do here. Hackathon time pressure is a real constraint. Pretending to have closed bias gaps that have not been closed is worse than naming them clearly.


### What the team did


Demographic aware extraction. The NLP demographic extractor reads age explicitly. The engine weights paediatric, adult, and geriatric scenarios differently. The four pre filled triage scenarios cover all three age bands.


Conservative defaults. When AI confidence is low or the engine cannot answer, severity falls back to moderate, not low. The reasoning is straightforward. Under triage is more dangerous than over triage in an emergency context.


Negation detection. The prompt and the post processor both try to catch phrases such as "no chest pain", "denies headache", and similar. Missing a negation is a classic source of clinical AI error.


No demographic inference from narrative. The model is not asked to infer race, gender identity, insurance type, or socioeconomic status from the symptom text. The system extracts clinical entities only.


### What the team did not do


No formal bias evaluation. The system was not run against a stratified evaluation set such as MIMIC-IV demographic splits. Performance differences across age, sex, or race were not measured. A production version would need to.


No translation. The system is English only. Patients whose first language is not English are at a disadvantage on the input side.


No accessibility audit. The user interface uses semantic HTML and keyboard shortcuts, but no WCAG audit was performed and the system was not tested with a screen reader.


The Gemini language model carries whatever biases it carries. Google does not publish bias statistics for Flash Lite at a level of detail that would allow a clinical use evaluation.


### What would close the gap


* A structured bias evaluation against an open clinical dataset such as MIMIC-IV with the appropriate data use agreement.
* Translation of the prompt and the input parser into the languages of the patient population the system would actually serve.
* A WCAG 2.2 AA accessibility audit of both the triage page and the console page.
* Clinical review of the inline guideline corpus by a practising clinician across multiple specialties.




## 4. Failure Cases


Every one of these failure modes is reachable from the demo. None of them silently corrupt a clinical decision.


| Failure | What happens | How the operator sees it | Where to verify in code |
| ------- | ------------ | ------------------------ | ----------------------- |
| Gemini API key missing | Tier 2 silently degrades to Tier 1, RAG and NLP only | Lower tier badge in the user interface | `services/ai-engine/tiered_ai.py` |
| Gemini quota exhausted with HTTP 429 | Same fallback path. The team swapped from `gemini-2.5-flash` to `gemini-2.5-flash-lite` specifically because of the free tier quota | Lower tier badge | `services/ai-engine/tiered_ai.py`, `services/ai-engine/main.py` |
| AI engine unreachable from the web tier | The Next.js proxy returns a Tier 3 response with a hard coded "Clinical review required" brief. The user interface still renders end to end | Amber Tier 3 badge with a banner | `apps/web/src/app/api/ai/analyze-intake/route.ts` |
| Schema drift in patient input, for example a numeric pain score where a string is expected | The Pydantic request model is permissive with `model_config = {"extra": "ignore"}` and severity is coerced from string, integer, or float so a numeric pain score does not blow up validation | Request still succeeds | `services/ai-engine/main.py` `_coerce_severity` |
| ICD-10 false positives from substring matching | Both extractors use `re.search` with word boundaries and a three character minimum on aliases. Without this fix, the substring matcher mapped "diaphoresis" to "suicidal ideation" via the substring "si" | Tag is simply not produced | `services/ai-engine/main.py` `_icd10_tag`, `_extract_medications` |
| Vital sign mis keying between display name and snake case key | `_vitals_to_dict` explicitly maps display names such as "BP Systolic" to the snake case keys used in `vitals_ranges.json` such as `bp_systolic`. It also converts Celsius to Fahrenheit before threshold comparison | Vitals correctly flagged as warning or critical | `services/ai-engine/main.py` `_vitals_to_dict` |
| Hallucinated medication or guideline reference | The language model is asked for structured JSON only and is told to never invent dosages. The post processor strips any field it does not recognise. The RAG card surfaces matched keywords so the operator can verify a guideline reference is real | Operator sees the matched keyword chips on the RAG card | `services/ai-engine/main.py` `_retrieve_guidelines` |




## 5. Safety Guardrails on the AI Output


The participant guide explicitly called out the difference between safe AI usage and unsafe AI usage in healthcare. The team tried to follow that guidance literally.


Disclaimers. Every AI generated brief on the triage page is rendered with a "for clinician review" footer. The disclaimer is not optional and not toggleable.


No clinical claims. The user interface does not present any AI output as a diagnosis. Suggested ICD-10 codes are explicitly labelled as candidates. Urgency is explicitly labelled as suggested. The FHIR `CarePlan` is explicitly labelled as a draft.


Grounding. The language model is given the patient narrative plus the top three RAG hits. It is told to base its response on what it has been given and to mark anything it is unsure about.


Structured output. The language model is asked for JSON, not prose. Structured output is significantly less prone to hallucination for clinical extraction tasks. This point is in the participant guide.


Provenance. Every field carries a `source_tier` and a `provenance` string so the operator can see exactly where a value came from.


No auto action on patient flow. The status state machine in the engine refuses to transition cases without a human action, regardless of what the AI suggested.




## 6. Regulatory Awareness


This is a hackathon prototype. It is not a 510 k submission. It is not an FDA cleared Software as a Medical Device. It is not HIPAA compliant in production form. It is not ONC certified. It has not been HL7 validated by an accredited body. It is not in any way authorised for use with live patients.


To productionise this in the United States the team would need at minimum the following changes.


| Area | What would need to change |
| ---- | ------------------------- |
| HIPAA | A signed Business Associate Agreement with the language model provider. Encryption at rest and in transit. Access logging at the FHIR resource level. A real audit store. The demo logs in memory only. |
| FDA Software as a Medical Device | A risk classification, likely Class II given the decision support scope. Independent clinical validation. 510 k or De Novo review. |
| ONC Information Blocking Rule | Real FHIR API endpoints. USCDI compliance. ONC certification of the wrapping platform. |
| EU MDR and EHDS | If shipped in Europe, a CE mark under the Medical Device Regulation and conformance with the European Health Data Space rules on data sharing. |
| Clinical validation | An IRB approved clinical study against a curated patient cohort, with sensitivity, specificity, and bias measurements reported in line with TRIPOD AI guidance. |


The team chose to be explicit about the gap rather than pretend it does not exist. The participant guide feasibility criterion specifically asks about regulatory awareness, and the team wanted to show its work.
